## 1. What's good

1. **Clear top-level layering.** [src/app/](src/app/) splits into `model`,
   `repository`, `data-source`, `config`, `messages`, `core`, `components` — each
   layer has a single responsibility and dependencies point one way
   (components → repository → data-source → http).
2. **Reusable generic base classes.**
   [core/repository/read-repository.ts](src/app/core/repository/read-repository.ts),
   [crud-repository.ts](src/app/core/repository/crud-repository.ts), and
   [core/table/common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts)
   capture CRUD + table editing once; per-entity classes collapse to a few lines.
3. **A real `core/` sub-tree** — pipes, directives, dialog helpers, drag,
   inline-edit, date/color utilities, plus the migration-era reusables
   (`nav-list-selector`, `AppDateAdapter`, shared form styles). Framework-agnostic
   and reused across siblings. The strongest part of the codebase.
4. **CQRS-lite separation.** Read and write repositories are decoupled and share a
   stream so a successful write triggers a reload everywhere.
5. **Centralized HTTP entry point.**
   [rest-data-source.ts](src/app/data-source/rest-data-source.ts) is the only place
   that knows REST URLs; [configuration.ts](src/app/config/configuration.ts) swaps
   the base URL via an `InjectionToken` + `fileReplacements`. Idiomatic.
6. **Modern Angular baseline** — standalone + zoneless + signals + typed forms +
   Angular Material, `@if`/`@for` control flow throughout.
7. **Test discipline.** Nearly every file has a `.spec.ts`; the pure-logic ones
   ([date-range-generator](src/app/core/utils/date-range-generator.spec.ts),
   [payment-utils](src/app/utils/payment-utils.spec.ts)) actually exercise behavior.

## 2. Still open

### 2.1 Repository class explosion
The many per-entity repository/token files are mostly a few lines of
configuration. A generic factory keyed by `resourceName` (or a single
`provideRepository<T>(name)` helper) could collapse most of them into data. This
also enables removing the last NgModule — see §2.8.

### 2.2 No HTTP interceptors / no typed error model
`provideHttpClient(withInterceptorsFromDi())` is wired, but **no interceptor
exists**. Errors are stringified into `MessagesService` (`'Network error:' + …`)
at each call site; there's no `catchError` funnel, no retry, no auth header, no
global loading indicator. One error+loading interceptor would remove duplicated
error handling and give consistent UX. (Consider renaming `MessagesService` →
`NotificationService` while at it.)

### 2.3 REST boundary: generics done, validation + `httpResource` still open
The compile-time `any` leak is fixed — `RestDataSource` is generic (`getResponse<T>`
etc.) and `ReadRepository<T>`/`CrudRepository<T>` carry `T` through to the signals
(see §0). Remaining gaps: **(a)** `T` is only an *asserted* type — nothing validates
the payload at runtime, so genuine backend schema drift still slips through silently;
**(b)** `postFormDataResponse(...)` still returns `HttpResponse<any>`; **(c)**
`httpResource<T>()` is not used anywhere — read-models could adopt it for a
signal-native fetch that drops the manual `toSignal`/`ReplaySubject` plumbing.

### 2.4 State is duplicated across repositories
Each repository holds its own data. Two components needing the same `PaymentRefs`
each trigger a fresh HTTP call. A shared signal store (plain `signal()`+`computed()`
or NgRx SignalStore) across `payments-master` ↔ `payments-table` ↔
`payments-summary` would remove redundant fetches and the reload choreography.

### 2.5 Heavy `payments-table` component
[payments-table.component.ts](src/app/components/payments-table/payments-table.component.ts)
is ~394 lines (down from ~500 pre-migration but still the largest): form builder,
inline-edit orchestration, validation, rounding, clipboard paste, previous-period
diffing. The private `static` math helpers (`roundValue*`) belong in
`core/utils/numeric.ts`; the inline-edit orchestration could be its own controller.

### 2.6 Constructor-based component inheritance taking a runtime constructor
[common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts)
accepts `ctor: new() => W`. This fights both DI and TS — composition (a factory
provider, or a `signal()`-based store) would be cleaner.

### 2.7 No lazy loading
[app.routes.ts](src/app/app.routes.ts) eagerly imports every feature component
(`component:` for `settings`, `payments/:id`, `reports/:id`, dashboard). Route-level
lazy loading is low-hanging fruit with standalone components:
`loadComponent: () => import('./…').then(m => m.SomeComponent)`. Cuts the initial
bundle (production build warns at ~2 MB).

### 2.8 Smoke-test debt
`RepositoryModule` is gone (§0). What remains is test *quality*: many `.spec.ts` only
assert `expect(component).toBeTruthy()` — they cost CI time without asserting
behavior. Extend the valuable pure-logic specs; trim the smoke ones. Two specs are
still `it.skip`-ed pending real fixes (exposed by the Karma→Vitest move):
[update-payment-group.component.spec.ts:31](src/app/components/update-payment-group/update-payment-group.component.spec.ts#L31)
and [data-management.component.spec.ts:31](src/app/components/data-managment/data-management.component.spec.ts#L31).

### 2.9 Bugs and smells
- `headers.append('Content-Type', undefined)` in
  [rest-data-source.ts:34](src/app/data-source/rest-data-source.ts#L34) — invalid;
  for `FormData` just don't set Content-Type.
- `Number.parseInt(x, 0)` in
  [payments-master.component.ts:27](src/app/components/payments-master/payments-master.component.ts#L27)
  — radix `0` accepts `0x…`/`0o…`; should be `10`.
- **`console.log` left in production paths (6):** the loading-signal traces in
  [read-repository.ts:73,101](src/app/core/repository/read-repository.ts#L73) and
  four `JSON.stringify` dumps in
  [common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts)
  (`Editing item`, `Delete PersistData`, `PersistData`, `Updating with`). Strip these.
- Typos: folder [components/data-managment/](src/app/components/data-managment/)
  (missing 'e'); method `pervPeriodCounterLabelClick`
  ([payments-table.component.ts](src/app/components/payments-table/payments-table.component.ts)).
  Rename in one PR (ripples across the selector/template).

### 2.10 README lacks orientation
[README.md](README.md) is a command scratchpad (the stale bootstrap/jQuery/FA
install notes were removed 2026-07). It still has **no architecture overview, no
domain glossary** (Payment / PaymentObject / PaymentGroup / PaymentRefs /
PaymentRep), and no docs on running the `server/` json-server. Also stale: the
"unit tests via Karma" / "e2e via Protractor" lines (now Vitest / Playwright) and
the Angular-CLI-8 header. Rewrite as: architecture diagram, glossary, run/deploy
steps; move upgrade history to `CHANGELOG.md` or git tags.

### 2.11 No CI / no local enforcement
Nothing runs lint + build + test on PRs (no `.github/workflows`). Add CI plus a
Prettier/ESLint check; optionally `husky` + `lint-staged` locally.

### 2.12 `@angular/animations` deprecation watch
`provideAnimations()` (app) and `provideAnimationsAsync('noop')` (test-setup) back
Angular Material's animations. The imperative animations package is on a
deprecation path (Angular plans removal ~v23); track Material's guidance and switch
to the successor API when available. (No longer tied to ngx-bootstrap — that's gone.)

### 2.13 Minor: D3 chart lifecycle
[reports-chart-date-totals.component.ts](src/app/components/reports-chart-date-totals/reports-chart-date-totals.component.ts)
still uses `ngAfterViewInit` + nested `setTimeout` to draw. It works (D3 writes the
SVG directly, no CD needed), but `afterNextRender` would be clearer and more
zoneless-idiomatic.

### 2.14 Full-table reload on every write — per-row update opportunity

Every successful CRUD/inline-PATCH routed through
[common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts)
`crudData$` calls `loadRepositoryData()`, refetching the whole list/`PaymentRefs`.
Desired UX: the affected row shows a spinner, **only that row visibly updates** on
success, full reload only on error.

> **Note (post-Material):** `payments-table` is now a **flex `mat-table`**, and
> during the migration it was changed to **hide the whole table while
> `loadingSignal()`** and rebuild `selectableItems` (new wrapper objects) on each
> load. So the flicker is real, and the fix differs slightly from the pre-migration
> `@for` advice.

**Recommended — cheap path (keep the reload):**
1. **Give `mat-table` a stable `trackBy`.** `<mat-table [dataSource]="selectableItems()">`
   currently uses the default identity trackBy, so new `SelectableItem` wrappers on
   each load recreate every row. Add `[trackBy]="trackById"` returning
   `row.value.id` so same-id rows reuse their DOM and only changed cells repaint.
   (The simple tables already key by id.)
2. **Per-row pending — component-level only.** A `pendingRowId` signal (or `Set`)
   set before `crudRepository.execute(...)` for Update/Delete/inline-Patch, cleared
   on the `crudData$` outcome. No repository changes.
3. **Row-loading in the template.** Revert the "hide the whole table while loading"
   gate; render a small spinner on the in-flight row instead.
4. **Errors** already trigger a full reload — leave as is.

**Heavier alternative — true per-row mutation (no reload):** back `dataSignal` with
a writable signal + an `update()` API; add an overridable `applyCrudResult()`
(default upsert/remove-by-id; custom for payments to replace the row + re-link
`prevPeriodPayment`); switch `CrudRepository`'s `switchMap` → `mergeMap`. **Verified
backend shape:** POST/PUT return the full DTO, DELETE returns empty, but the inline
PATCH (`PaymentController.partialUpdate`) returns `RowsAffectedDTO` — so the
no-reload path must apply the client-sent value for inline edits. Reserve this for
when the round-trip itself is a real problem (large lists / high write frequency).

**Caveat:** with the current `switchMap`, rapid back-to-back row edits serialize
(a second op cancels the first's request). Fine for one-cell-at-a-time inline
editing; revisit `mergeMap` only for genuinely concurrent edits.

### 2.15 `withHashLocation()` workaround for Tomcat 404s
[app.config.ts](src/app/app.config.ts) uses `provideRouter(routes, withHashLocation())`.
Reason: deep-linking to `/rainments-ang/payments/123` on Tomcat 404s (Tomcat looks
for a static file). Hash routing (`/#/payments/123`) sidesteps this since the
server never sees the fragment. **Constraint:** server-level Tomcat config isn't
always available (shared/managed instances).

**Costs:** ugly, unshareable URLs; the fragment is unavailable for in-page anchors;
server logs show only `/`; SSR/prerender/SEO impossible; `#` confuses external
tools (email, ticketing, link previews).

**Deployment topology:** `rainments-ang` is `ng build --base-href=/rainments-ang/`
deployed as **exploded static files** to `<tomcat>/webapps/rainments-ang/` (not a
WAR, but Tomcat honors a `WEB-INF/web.xml` placed there). `rainments-wss` is a
Spring Boot WAR at a separate context; CORS bridges them today.

#### Option A — minimal: add `WEB-INF/web.xml` to the frontend deploy dir
Keep two deployables. Add `rainments-ang/webapp-templates/WEB-INF/web.xml` with a
404 → `/index.html` `<error-page>`, have the deploy script
([common/builder/builder.psm1](../common/builder/builder.psm1) `Start-DeployWeb`)
copy the `WEB-INF` folder alongside the static files, then drop `withHashLocation()`.
*Caveat:* HTTP status stays 404 (body is `index.html`, so browsers render fine, but
monitoring/CDN layers may misread). Returning 200 needs a servlet filter → a Java
build step in the pure-static frontend → not worth it; use Option B if the status
code matters. **~half-day; backend untouched.**

#### Option B — cleaner: serve the SPA from the Spring Boot WAR
Output Angular `dist/` into Spring's static path (gradle `Copy` task in
`rainments-wss`), adjust `--base-href`, add a `@Bean ErrorPageRegistrar`
(404 → `/index.html`; the Spring Boot equivalent of `<error-page>` — **not** a
hand-written `web.xml`, because `bootWar` uses the `BOOT-INF/classes/static/`
layout), drop CORS from
[WebConfig.java](../rainments-wss/src/main/java/com/romanpulov/rainmentswss/config/WebConfig.java),
drop `withHashLocation()`, and skip the separate frontend deploy.
**URL-collision check:** backend has no `GET /payments/{id}` (only PUT/DELETE), so
`GET /rainments-wss/payments/123` → no controller → 404 → SpaConfig forwards to
`/index.html` → Angular routes to `/payments/123` ✓, while its AJAX `GET /payments`
list still hits the controller. New `GET /...  /{id}` backend routes would need an
`/api/` prefix to stay collision-safe. **~half-day + testing; one deployable, no CORS.**

**Recommendation:** Option A for the smallest change; **Option B** if you'll invest
the half-day — the backend already has the right shape and CORS is itself a
workaround, making B the more honest end state. *Fallback* (no `WEB-INF/` access): a
service worker serving `index.html` on navigate-fetch failure — only helps after
the SW is installed (first-visit cold deep-links still 404); not a real substitute.
