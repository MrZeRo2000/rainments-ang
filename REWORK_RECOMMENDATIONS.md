## 1. What's good

1. **Clear top-level layering.** [src/app/](src/app/) splits into `model`, `repository`, `data-source`, `config`, `messages`, `core`, `components` â€” each layer has a single responsibility and dependencies point one way (components â†’ repository â†’ data-source â†’ http).
2. **Reusable generic base classes.** [core/repository/read-repository.ts](src/app/core/repository/read-repository.ts), [read-write-repository.ts](src/app/core/repository/read-write-repository.ts), and [core/table/common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts) capture CRUD + table editing patterns once. Per-entity classes like [payment-repository.ts](src/app/repository/payment-repository.ts) collapse to a few lines.
3. **A real `core/` sub-tree.** Pipes, directives, modal helpers, drag, inline-edit, date utilities, color utilities â€” all framework-agnostic and reusable across siblings. This is genuinely the strongest part of the codebase.
4. **CQRS-lite separation.** Read repositories and persist repositories are decoupled (`*-repository` + `*-persist-repository`), sharing a `Subject` so a successful write triggers a reload everywhere.
5. **Centralized HTTP entry point.** [rest-data-source.ts](src/app/data-source/rest-data-source.ts) is the only place that knows about REST URLs; [configuration.ts](src/app/config/configuration.ts) uses an `InjectionToken` swapped at build-time via `fileReplacements`. Idiomatic.
6. **Test discipline.** Nearly every file has a `.spec.ts` companion. The pure-logic ones ([date-range-generator](src/app/core/utils/date-range-generator.spec.ts), [payment-utils](src/app/utils/payment-utils.spec.ts)) actually exercise behavior.
7. **Template uses modern `@if`/`@for`** control flow consistently.

### 2.8.1 `withHashLocation()` workaround for Tomcat 404s
[app.config.ts](src/app/app.config.ts) uses `provideRouter(routes, withHashLocation())`. The reason (per developer): deep-linking to a routed path like `/rainments-ang/payments/123` on Tomcat returns 404, because Tomcat looks for a static file at that path. Hash routing sidesteps this by encoding the route in the URL fragment (`/#/payments/123`), which the server never sees. **Constraint:** server-level Tomcat configuration is not always available (shared / managed instances).

**Costs of the current workaround:**
- Ugly URLs (`/#/...`); not friendly to share/copy
- The URL fragment is unavailable for in-page anchors (router owns it)
- Server access logs don't show the route, only `/`
- SSR / prerender / SEO is essentially impossible
- The `#` confuses many external tools (e-mail clients, ticketing systems, link previewers)

**Deployment topology (as of 2026-05-29):**
- **`rainments-ang`** is built via `ng build --base-href=/rainments-ang/` and deployed as **exploded static files** to `<tomcat>/webapps/rainments-ang/` (via `common/builder/builder.psm1::Start-DeployWeb`). It is *not* packaged as a WAR file, but Tomcat treats any directory under `webapps/` as a webapp â€” so you CAN place a `WEB-INF/web.xml` inside that directory and Tomcat will honor it.
- **`rainments-wss`** is a Spring Boot WAR (`bootWar`) deployed to `<tomcat>/webapps/rainments-wss.war`. Frontend and backend live at separate context paths today; CORS is enabled in [WebConfig.java](../rainments-wss/src/main/java/com/romanpulov/rainmentswss/config/WebConfig.java) to bridge them.

Given that topology, there are two real options to retire `withHashLocation()`:

---

#### Option A â€” minimal: add `WEB-INF/web.xml` to the frontend's deployment directory

Keep the current two-deployable architecture. Just add a `web.xml` template to `rainments-ang/` and have the deploy script copy it into `<tomcat>/webapps/rainments-ang/WEB-INF/` alongside the static files.

**Steps:**

1. Create `rainments-ang/webapp-templates/WEB-INF/web.xml`:
   ```xml
   <?xml version="1.0" encoding="UTF-8"?>
   <web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
            xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
            xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee
                                https://jakarta.ee/xml/ns/jakartaee/web-app_6_0.xsd"
            version="6.0">
     <error-page>
       <error-code>404</error-code>
       <location>/index.html</location>
     </error-page>
   </web-app>
   ```
   (Tomcat 9.x: use the older `http://xmlns.jcp.org/xml/ns/javaee` namespace and `version="4.0"`. The `<error-page>` element itself is stable since Servlet 2.3.)

2. Update [common/builder/builder.psm1](../common/builder/builder.psm1) `Start-DeployWeb` to also copy the WEB-INF folder:
   ```powershell
   $WebInfSrc = "$PSScriptRoot../../../$project/webapp-templates/WEB-INF"
   if (Test-Path $WebInfSrc) {
     Copy-Item -Path $WebInfSrc -Destination "$DestinationFolder\" -Recurse -Force
   }
   ```

3. Drop `withHashLocation()` from `app.config.ts`: `provideRouter(routes)`.

**Caveat â€” HTTP status remains 404** (body is `index.html`, so browsers render correctly, but monitoring/CDN layers may misinterpret). To return 200, a tiny servlet filter would need to be packaged with the frontend deploy â€” which means introducing a Java build step into the otherwise pure-static frontend project. Likely not worth it; if status code matters, prefer Option B.

**Pros:** Minimal change, backend WAR untouched, half-day of work.
**Cons:** Frontend gains a hidden Java-EE artifact; 404 status not fixable without introducing a Java compile step into the frontend.

---

#### Option B â€” cleaner: merge the frontend into the Spring Boot WAR

Long-term better: serve the Angular app from the same Spring Boot WAR as the API. Spring Boot has first-class SPA support and the Tomcat plumbing is already there.

**Steps:**

1. **In `rainments-ang`:** change the build to output Angular's `dist/` into Spring's static path. Either set Angular's `outputPath` in `angular.json`, or add a gradle copy task in `rainments-wss/build.gradle`:
   ```gradle
   def angularDist = file("../rainments-ang/dist/rainments-ang/browser")

   tasks.register('copyAngular', Copy) {
       from angularDist
       into file("src/main/resources/static")
       onlyIf { angularDist.exists() }
   }
   processResources.dependsOn copyAngular
   ```
   Add `src/main/resources/static/` to `.gitignore`.

2. **Adjust the base-href** in the Angular build command (in `deployWebProject.ps1`) from `--base-href=/rainments-ang/` to `--base-href=/rainments-wss/` (or whatever the combined context becomes). Or strip the trailing prefix entirely if the combined WAR is deployed as ROOT.

3. **Add `SpaConfig.java`** in `rainments-wss/src/main/java/com/romanpulov/rainmentswss/config/`:
   ```java
   @Configuration(proxyBeanMethods = false)
   public class SpaConfig {
       @Bean
       public ErrorPageRegistrar spaErrorPageRegistrar() {
           return registry -> registry.addErrorPages(
               new ErrorPage(HttpStatus.NOT_FOUND, "/index.html")
           );
       }
   }
   ```
   This is the Spring Boot equivalent of `<error-page>` in `web.xml`. **Same 404-status caveat applies.** A path-regex-based fallback `@Controller` could return 200, but maintaining the exclusion list (`/payments`, `/payment-objects`, `/payment-groups`, `/products`, `/app:*`, `/app-info`, `/hello`) is brittle â€” keep the `ErrorPageRegistrar` unless the 404 status hurts monitoring.

4. **Drop CORS** from [WebConfig.java](../rainments-wss/src/main/java/com/romanpulov/rainmentswss/config/WebConfig.java) â€” no longer needed when frontend and backend share an origin.

5. **Drop `withHashLocation()`** from `app.config.ts`.

6. **Update deployment scripts** to skip the separate frontend deploy step.

**URL-collision check** â€” backend already has `GET /payments`, `GET /payment-groups`, `GET /payment-objects`, `GET /products`, `GET /app-info`, `GET /hello`, `GET /app:*`, plus `{PUT,DELETE}/{id}` per-entity. The Angular route `/payments/:id` does NOT collide with any backend `GET /payments/{id}` handler (none exists â€” only PUT/DELETE on that path). So `GET /rainments-wss/payments/123` â†’ no controller match â†’ 404 â†’ SpaConfig forwards to `/index.html` â†’ Angular boots â†’ routes to `/payments/123` âś“. Then Angular's own AJAX calls go to `GET /payments` (the list), which IS a controller â€” works normally. The architecture is collision-safe today, but new backend `GET /payments/{id}`-style routes would need to be added carefully (or moved under an explicit `/api/` prefix).

**Pros:** One deployable; no CORS; Java-side configurability for SPA fallback, auth integration, etc.; standard Spring Boot pattern.
**Cons:** ~half-day to wire up; frontend and backend deploys become coupled; Spring Boot's executable WAR layout puts static at `BOOT-INF/classes/static/` (handled automatically by `processResources`, but worth verifying).

---

#### Important distinction: plain WAR vs Spring Boot executable WAR

The `bootWar` plugin produces a **Spring Boot executable WAR** with non-standard layout (`BOOT-INF/classes/static/` for resources instead of the root). Do NOT add a `<error-page>` to a hand-written `WEB-INF/web.xml` inside the Spring Boot WAR â€” use `ErrorPageRegistrar` instead (that's the bean Spring Boot translates into the equivalent server-level registration). Plain-WAR `web.xml` works for the **frontend exploded directory** (Option A) because that's not a Spring Boot artifact.

---

#### Recommendation

- **Option A** if the two-deployable architecture is fine and you want the smallest possible change.
- **Option B** if you're willing to invest a half-day to simplify the deployment model long-term. The fact that the backend already has the right shape and your CORS is currently a workaround makes Option B the more honest answer to "what should this look like."

**Fallback if neither option is workable** (e.g., dropping pure-static files into `webapps/ROOT/` with no `WEB-INF/` access): a service worker that serves `index.html` for `navigate` fetch failures. Only helps on the 2nd visit (SW must be installed first); first-visit cold-start deep links still 404. Not a real substitute.

### 2.15 Sass `@import` is deprecated (will be removed in Dart Sass 3.0.0)
[src/styles.scss](src/styles.scss) uses three `@import`s. Two are upstream (`bootstrap/scss/bootstrap`, `ngx-bootstrap/datepicker/...`) and one is own code (`_variables.scss`). Recommended migration:

1. **Convert the own-code import:** `@import 'variables';` â†’ `@use 'variables';` (variables become namespaced as `variables.$primary`, etc.).
2. **Convert the Bootstrap import to `@use ... with (...)`** â€” Bootstrap 5.3.8 (currently installed) supports the module syntax properly:
   ```scss
   @use 'bootstrap/scss/bootstrap' as bootstrap with (
     $primary: variables.$primary,
     $tooltip-arrow-color: variables.$tooltip-arrow-color,
     $tooltip-bg: variables.$tooltip-bg,
     $tooltip-color: variables.$tooltip-color
   );
   ```
   This replaces the old "define `!default`-overrides before importing" timing trick with an explicit override list.
3. **Namespace-prefix Bootstrap variable references** in the rest of `styles.scss`: `$border-radius` â†’ `bootstrap.$border-radius`, `$table-cell-padding-y`, `$table-border-color`, `$btn-border-radius` (lines 89, 90, 100, 101, 121 of the current file).
4. **`node_modules/` path prefix is optional** â€” Angular's Sass loader has `node_modules` on its load path; `@use 'bootstrap/scss/bootstrap'` resolves the same as the longer form.
5. **ngx-bootstrap stays as `@import`** â€” `ngx-bootstrap` 19.x still uses `@import` internally; its deprecation warning is upstream-controlled. **Leave it visible** (per project convention) â€” when it disappears, that's the signal upstream has migrated and you can `@use` it too.
6. **Possibly-dead variable:** `$fa-font-path` in [_variables.scss](src/_variables.scss) â€” grep before keeping; no current SCSS file imports the FontAwesome SCSS, so it may be orphaned.

The automated migrator (`npx sass-migrator module src/styles.scss`) does step 1 mechanically. Steps 2-3 require manual care because they change variable-resolution semantics.

### 2.16 Standalone-component DI gotcha: pipes used via `inject()`
A standalone component that uses a pipe **only in its template** needs the pipe class in `imports: [...]`. A component that **injects the pipe class** (e.g., `private p = inject(AmountPipe)` for programmatic formatting) needs the pipe additionally in `providers: [...]`. Under the old NgModule setup, declaring a pipe in `CoreModule.declarations` covered both cases â€” with standalone, you opt in per component.

Current state: only [payments-table.component.ts:87](src/app/components/payments-table/payments-table.component.ts#L87) does this (`inject(AmountPipe)`); fixed 2026-05-29 by adding `providers: [AmountPipe]`. If similar `inject(...Pipe)` patterns are introduced later, remember this rule â€” the error is `NG0201: No provider found for _XxxPipe`.

### 2.17 Full-table reload on every write â€” per-row update opportunity

**Context (post-signals-migration).** Every successful CRUD or inline PATCH routed through [common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts) `crudData$` calls `loadRepositoryData()` â€” refetching the whole list (or the whole `PaymentRefs` composite) and re-running its post-processing (e.g. the `mapPaymentRefs` prev-period re-link + summary recompute). The desired UX is: the row being updated/deleted shows a loading indicator, **only that row visibly updates** on success, and a full reload happens **only on error**.

There are two ways to get there; **the cheap one is recommended.**

#### Recommended â€” keep the reload, lean on `@for â€¦ track <id>`

Angular's `track` keys rows by identity: a same-`id` row reuses its existing `<tr>` DOM node and only the changed cell bindings repaint; new/removed ids add/remove rows. So **even with a full refetch, unchanged rows don't flicker and only the changed row updates visibly** â€” which is the actual goal. Note `track` does not skip the HTTP call and does not content-diff; it controls DOM-node reuse, and normal change detection handles the per-binding update.

Work required (small, no core-repository rework):
1. **Fix track keys.** [payments-table.component.html](src/app/components/payments-table/payments-table.component.html) still tracks the wrapper object â€” `@for (item of selectableItems(); track item)` â†’ `track item.value.id`. The rows are rebuilt as new `SelectableItem` wrappers on each load, so object-identity tracking recreates every row; tracking the underlying `Payment.id` reuses them. (The simple tables already use `track item.id`.)
2. **Per-row pending â€” component-level only.** Add a `pendingRowId` signal (or small `Set`); set it right before `crudRepository.execute(...)` for Update/Delete/inline-Patch, clear it on the `crudData$` outcome. No `CrudRepository`/`ReadRepository` change.
3. **Row-loading in the template.** `@if (isRowPending(item)) { <app-core-row-loading [colspan]="â€¦"/> } @else { â€¦rowâ€¦ }`, backed by a small shared `CoreRowLoadingComponent` (reuse `LoadingSpinnerElementComponent`). For payments-table, **revert the "hide the whole table while `loadingSignal()`" gate** back to showing the table with a per-row spinner.
4. **Errors** already trigger a full reload â€” leave as is.

Cost: one HTTP round-trip + read-repo post-processing per write (negligible at these table sizes). Meets all four goals with the least code and risk.

#### Heavier alternative â€” true per-row mutation (no reload)

Avoids the round-trip entirely, at materially more code/risk:
- `ReadRepository`: back `dataSignal` with a writable signal and add an `update((current) => next)` mutation API.
- `CommonEditableTableComponent`: an overridable `applyCrudResult(action, result)` â€” default upsert/remove-by-id for simple `R = W` tables; a custom one for payments that replaces the row in `paymentList`, re-links `prevPeriodPayment`, and lets the `computed` summary recompute.
- `CrudRepository`: correlate each emitted result with its originating action, and switch `switchMap` â†’ `mergeMap` so independent rows can be in flight at once.

**Backend response shapes (verified against `rainments-wss`):** the generic `AbstractServiceRestController` POST/PUT return the full DTO and DELETE returns an empty body; **the payments inline PATCH (`PaymentController.partialUpdate`) returns `RowsAffectedDTO`, not the `Payment`.** So the no-reload variant must apply the value the *client* sent for inline edits (it cannot rebuild the row from the response). The track-id variant sidesteps this because it refetches.

#### Recommendation

Do the **track-id + per-row-loading** version. Reserve the per-row-mutation rework for when avoiding the reload round-trip itself becomes a real requirement (large lists, high write frequency, or offline-ish UX).

**Caveat:** with the current `switchMap` in `CrudRepository`, rapid back-to-back row edits serialize â€” a second op cancels the first's request. Fine for one-cell-at-a-time inline editing; only revisit `mergeMap` + resultâ†”action correlation if genuinely concurrent row edits are needed.
