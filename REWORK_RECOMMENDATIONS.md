## 1. What's good

1. **Clear top-level layering.** [src/app/](src/app/) splits into `model`, `repository`, `data-source`, `config`, `messages`, `core`, `components` â€” each layer has a single responsibility and dependencies point one way (components â†’ repository â†’ data-source â†’ http).
2. **Reusable generic base classes.** [core/repository/read-repository.ts](src/app/core/repository/read-repository.ts), [read-write-repository.ts](src/app/core/repository/read-write-repository.ts), and [core/table/common-editable-table-component.ts](src/app/core/table/common-editable-table-component.ts) capture CRUD + table editing patterns once. Per-entity classes like [payment-repository.ts](src/app/repository/payment-repository.ts) collapse to a few lines.
3. **A real `core/` sub-tree.** Pipes, directives, modal helpers, drag, inline-edit, date utilities, color utilities â€” all framework-agnostic and reusable across siblings. This is genuinely the strongest part of the codebase.
4. **CQRS-lite separation.** Read repositories and persist repositories are decoupled (`*-repository` + `*-persist-repository`), sharing a `Subject` so a successful write triggers a reload everywhere.
5. **Centralized HTTP entry point.** [rest-data-source.ts](src/app/data-source/rest-data-source.ts) is the only place that knows about REST URLs; [configuration.ts](src/app/config/configuration.ts) uses an `InjectionToken` swapped at build-time via `fileReplacements`. Idiomatic.
6. **Test discipline.** Nearly every file has a `.spec.ts` companion. The pure-logic ones ([date-range-generator](src/app/core/utils/date-range-generator.spec.ts), [payment-utils](src/app/utils/payment-utils.spec.ts)) actually exercise behavior.
7. **Template uses modern `@if`/`@for`** control flow consistently.

## 2. What's not so good

### 2.1 Outdated Angular patterns despite v21 dependencies
- Every component carries `standalone: false`. No standalone components, no `inject()`, no `signal()` / `computed()` / `effect()`, no `httpResource`, no typed forms, no new Router APIs.
- ZoneJS is still in use; templates call methods like `getSelectablePayments()` and `getPrevPeriodProductCounter(...)` on every CD cycle, with no `OnPush`.
- `UntypedFormBuilder` / `UntypedFormGroup` used throughout [payments-table.component.ts:111-117](src/app/components/payments-table/payments-table.component.ts#L111-L117) â€” typed forms (Angular 14+) would catch the dozens of string-based `controls.product.value` accesses.

### 2.2 Empty / vestigial modules
- [model/model.module.ts](src/app/model/model.module.ts) â€” empty NgModule, declares nothing.
- [config/config.module.ts](src/app/config/config.module.ts) â€” exists only to provide one token. Both should be deleted; the token should be `providedIn: 'root'`.

### 2.3 Repository class explosion
About 40 repository files. Most `*-persist-repository.ts` are 6-line constructors (see [payment-persist-repository.ts](src/app/repository/payment-persist-repository.ts)). A generic factory keyed by `resourceName`, or a single `provideRepository<T>(name)` helper, could collapse ~80% of these into configuration data.

### 2.4 Leaky reactive plumbing
- `ReadRepository.data: T[]` is mutated in place via `splice` ([read-repository.ts:70-74](src/app/core/repository/read-repository.ts#L70-L74)); consumers must poll via `getData()`.
- `loadSuccess`, `persistSuccess`, `loadingState` are exposed as raw `Subject<T>` ([persist-repository.ts:18-28](src/app/core/repository/persist-repository.ts#L18-L28)) â€” consumers can call `.next()` on them. Should be `asObservable()`, and for `loadingState` a `BehaviorSubject` (late subscribers miss the loading=true edge).
- Subscriptions inside repositories ([read-write-repository.ts:19](src/app/core/repository/read-write-repository.ts#L19)) are never torn down.

### 2.5 No HTTP interceptors / no typed error model
Errors are stringified and shoved into `MessagesService` (`'Network error:' + ...`). No `catchError`, no retry, no auth header injection, no global loading interceptor. Adding one interceptor would eliminate dozens of repeated error blocks.

### 2.6 Heavy components doing too much
[payments-table.component.ts](src/app/components/payments-table/payments-table.component.ts) is ~500 lines: form builder, inline-edit orchestration, validation, rounding, clipboard paste, color emission, and ad-hoc previous-period diffing. Several private `static` math helpers belong in `core/utils`.

### 2.7 Constructor-based component inheritance taking a runtime constructor
[common-editable-table-component.ts:25](src/app/core/table/common-editable-table-component.ts#L25) accepts `ctor: new() => W`. This pattern fights both DI and TS â€” composition (a `useFactory` provider, or a `signal()`-based store) would be cleaner.

### 2.8 No lazy loading
[app-routing.module.ts](src/app/app-routing.module.ts) eagerly imports every feature component. With the production build warning at 2MB initial bundle, route-level lazy loading is low-hanging fruit. Also: `useHash: true` is set with no apparent reason.

### 2.9 State is duplicated across repositories
Each repository holds its own `data: T[]`. Two components needing the same `PaymentRefs` each trigger a fresh HTTP call. A signal store (or NgRx Component Store) shared across `payments-master` â†” `payments-table` â†” `payments-summary` would remove redundant fetches and the `getLoadSuccessObservable` choreography.

### 2.10 Weak/no type safety at REST boundary
`getResponse(...)`, `postResponse(...)` all return `Observable<HttpResponse<any>>`. `any` leaks into every repository and out through `getData()`. A typed `RestDataSource<T>` (or `httpResource<T>()`) would surface schema drift at compile time.

### 2.11 Bugs and smells
- `headers.append('Content-Type', undefined)` in [rest-data-source.ts:37](src/app/data-source/rest-data-source.ts#L37) â€” invalid header; for `FormData` you just don't set Content-Type.
- `Number.parseInt(x, 0)` used widely (e.g. [payments-master.component.ts:26](src/app/components/payments-master/payments-master.component.ts#L26)) â€” radix `0` accepts `0x...` and `0o...`; should be `10`.
- `console.log('load data from ' + ...)` ([read-repository.ts:56](src/app/core/repository/read-repository.ts#L56)) and `console.log('constructing PersistRepository')` left in production code paths.
- Typo: folder [components/data-managment/](src/app/components/data-managment/) (missing 'e'); method `pervPeriodCounterLabelClick` ([payments-table.component.ts:486](src/app/components/payments-table/payments-table.component.ts#L486)).

### 2.12 Tests mostly verify "the framework still works"
Most `.spec.ts` files only check `expect(component).toBeTruthy()` while pulling in the whole `RepositoryModule` + `CoreModule`. They cost CI minutes without asserting behavior. The valuable specs (`date-range-generator`, `payment-utils`, `payments-color-utils`) prove the pattern works â€” extend it, drop the smoke tests.

> **Note:** the Karma â†’ Vitest migration exposed that several of these smoke tests were not just trivial but partially broken (they passed only because Karma's loose defaults silenced unknown-element/property/iterator errors). Two have been marked `it.skip` pending real fixes.

### 2.13 Dead/legacy tooling
- ~~Protractor + `app.e2e-spec.ts`~~ â€” replaced with Playwright on 2026-05-27.
- ~~Karma + Jasmine~~ â€” replaced with Vitest on 2026-05-27.
- `karma-cli` was sitting in `dependencies` instead of `devDependencies` (removed in same migration). `webpack` is still in `dependencies` and should move to `devDependencies` if needed at all.

### 2.14 README is a personal scratchpad
[README.md](README.md) mixes "install bootstrap 5" notes, version upgrade commands, and migration breadcrumbs. There's no architecture overview, no domain glossary (Payment / PaymentObject / PaymentGroup / PaymentRefs / PaymentRep), no docs on running `server/`. New contributors would be lost.

## 3. Proposed improvements, ranked by leverage

| # | Change | Why |
|---|--------|-----|
| 1 | **Migrate to standalone + signals + typed forms incrementally.** Start at leaves (pipes, simple components). | Unlocks every other modernization. Removes 90% of NgModule boilerplate. Also makes the per-spec `import '../../app.module';` workaround installed during the Vitest migration unnecessary. |
| 2 | **Replace `*-persist-repository.ts` boilerplate with one generic `createRepository<T>(name)` factory.** | Deletes ~30 trivial files; new entities become one-liners. |
| 3 | **Introduce an `HttpInterceptor` for errors + loading.** Funnel into a single `NotificationService` (rename `MessagesService`). | Removes duplicated `catchError`/message code from every repository; consistent UX. |
| 4 | **Add route-level lazy loading** for `/payments/:id` and `/reports/:id`. Drop `useHash: true` if not required. | Cuts initial bundle, easier code-splitting. |
| 5 | **Adopt a shared signal-based store** (Angular `signal()` + `computed()`, or NgRx SignalStore) for `PaymentRefs` and `PaymentRep`. | Removes the `loadSuccess` subject choreography and double-fetching. |
| 6 | **Type the REST surface.** `RestDataSource.get<T>(...)` and `httpResource<T>()` for read-models. Generate types from the backend if possible. | Compile-time guarantees instead of `any`. |
| 7 | **Replace `Subject` with `BehaviorSubject` / `signal`** for `loading` and `loadingState`; expose via `asObservable()` / readonly signals. | Fixes the late-subscriber bug and the "subjects leak `.next()`" issue. |
| 8 | **Split `payments-table.component.ts`** into: table view, inline-edit controller, form host, summary host. Move static math helpers to `core/utils/numeric.ts`. | The 500-line file is the project's biggest comprehension cost. |
| 9 | **Delete dead modules** (`ModelModule`, `ConfigModule`) and the trivial `toBeTruthy()` smoke specs. Convert remaining specs to assert real behavior; un-skip the two `it.skip`-ed specs by adding proper input/dependency mocks. | Less to maintain; signal of intent. Also gets test count back to honest. |
| 10 | ~~Replace Protractor with Playwright; replace Karma with Vitest.~~ | **Done 2026-05-27.** |
| 11 | **Strip `console.log`** and fix small bugs: `parseInt(..., 10)`, the `Content-Type: undefined` line, the `data-managment` / `pervPeriod` typos (do these in a single rename PR). | Hygiene; the typo rename has a ripple across selectors so worth doing once. |
| 12 | **Rewrite the README** as: architecture diagram, domain glossary, how to run the json-server, deploy steps. Move upgrade history into a `CHANGELOG.md` or git tags. | Lowers onboarding cost considerably. |
| 13 | **Add CI** (lint + build + test) and a Prettier/ESLint config check on PR. Add `husky` + `lint-staged` for local enforcement. | Currently nothing enforces consistency. |
| 14 | **Reconsider `webpack` in `dependencies`** â€” move to `devDependencies` or remove. | Trims install size, reflects intent. |
| 15 | **Watch the `@angular/animations` deprecation.** `provideAnimationsAsync` (used in `src/test-setup.ts` to support ngx-bootstrap modals under tests) is deprecated; Angular plans to remove the imperative animations package in v23. Track ngx-bootstrap's migration or replace it. | Avoids a surprise upgrade blocker. |
